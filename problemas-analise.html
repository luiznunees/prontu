🔍 Análise de Problemas e Inconsistências

Analisei todos os arquivos do projeto e identifiquei vários pontos que podem causar problemas em produção.

⚠️ 1. Lógica de "1 trabalho grátis por mês" quebrada

Problema: O código ainda referência campos que foram removidos do banco.

Arquivo: src/components/FormularioTrabalho.tsx:97

if (profile?.plano === "gratis" && profile?.trabalhos_gerados_mes >= 1) { setIsPayModalOpen(true); }

Problema:

  • Campo plano foi removido na migração 002_credits.sql
  • Campo trabalhos_gerados_mes foi removido na mesma migração
  • Agora o sistema usa creditos diretamente
  • Essa condição nunca será verdadeira porque esses campos não existem mais

O que acontece na prática:

Usuários logados podem gerar trabalhos indefinidamente desde que tenham créditos. O limite "1 por mês" não existe mais no código, mas a UI ainda promete isso ("1 trabalho grátis por mês" no landing e no modal de auth).

⚠️ 2. Lógica de Watermark incorreta

BUG CRÍTICO! A marca d'água quase NUNCA será adicionada.

Arquivo: src/app/api/gerar-pdf/route.ts:78

// Marca d'água: apenas no crédito inicial gratuito (creditos === 1) // Se tem mais de 1 crédito, é porque comprou — sem marca d'água const addWatermark = creditos === 1;

Problema:

  • Quando usuário compra 5 créditos, começa com 5
  • Após gerar 1º trabalho: 5 → 4 (não é === 1, sem marca d'água)
  • Após gerar 2º trabalho: 4 → 3 (não é === 1, sem marca d'água)
  • Após gerar 5º trabalho: 1 → 0 (AGORA é === 1, ADICIONA marca d'água!)

Resultado: O último trabalho (quando créditos estão acabando) que leva marca d'água, não o primeiro!

Solução correta:

Adicionar campo primeiro_trabalho_feito na tabela users para rastrear se o usuário já usou seu crédito gratuito.

⚠️ 3. Usuários novos podem começar com 0 créditos

BUG! Na criação do usuário, não há atribuição de créditos iniciais.

Arquivo: src/lib/auth-server.ts:21-39

export async function createUserIfNotExists(email: string, nome: string, id: string) { const { data, error } = await supabase .from('users') .insert([{ id, email, nome, plano: 'gratis' }]) // ❌ plano não existe + sem creditos! .select() .single(); }

Problema:

  • Na migração 002, creditos tem DEFAULT 1
  • MASS o insert explicitly não passa creditos
  • Em alguns casos (race conditions, banco fresh), pode iniciar como NULL ou 0
  • Usuário pode tentar gerar trabalho e já estar bloqueado

Recomendação:

.insert([{ id, email, nome, plano: 'gratis', // ❌ remove isso creditos: 1 // ✅ adiciona isso }])

⚠️ 4. Webhook não é verdadeiramente idempotente

Risco: Se o webhook for chamado 2x, usuário recebe créditos dobrados.

Arquivo: src/app/api/pagamento/webhook/route.ts

// Verificar se já foi processado (idempotência) const { data: pagamento } = await supabase .from("pagamentos") .select("status, creditos_comprados") .eq("pix_id", pixId) .single(); if (pagamento?.status === "PAID") { return NextResponse.json({ received: true }); } // ... aí adiciona créditos await adicionarCreditos(userId, creditosAdicionar);

Problema:

  • Existe verificação, MAS se falhar entre o check e o update, não há rollback
  • Se webhook der timeout na linha 47 (adicionarCreditos), mas a verificação passou, não retry
  • Sem Transactions.atomicidade

Cenário de falha:

  1. Webhook recebe evento PIX pago
  2. Verifica status = PENDING ✅
  3. Tenta adicionar créditos... TIMEEOUT
  4. Webhooks não têm retry automático no AbacatePay (configurável mas não usado)
  5. Usuário PAGOU mas não recebeu créditos

⚠️ 5. Sem rate limiting na API de gerar PDF

Risco de abuse: Qualquer um pode spammar a API e gastar créditos do usuário logado.

Arquivo: src/app/api/gerar-pdf/route.ts

Não há:

  • Rate limit por IP
  • Rate limit por usuário
  • Circuit breaker
  • Timeout nos serviços externos (Gemini, geração de capa)

Cenário:

  • Usuário malicioso (ou bug no frontend) faz 100 requests simultâneas
  • Cada uma consome 1 crédito
  • Usuário perde todos os créditos em segundos

⚠️ 6. Políticas RLS podem bloquear criação de usuários

Potential blocker: Usuários novos podem falhar no login.

Arquivo: supabase/migrations/001_initial.sql:39-40

CREATE POLICY "Users read own" ON users FOR SELECT USING (auth.uid() = id); CREATE POLICY "Users update own" ON users FOR UPDATE USING (auth.uid() = id); -- ❌ NÃO EXISTE: INSERT policy!

Problema:

  • Quando usuário faz login pela 1ª vez via Google OAuth
  • O callback tenta fazer INSERT na tabela users
  • Se as policies default.blockanônimas estiver ativa, pode falhar
  • Porém, como usa SECURITY DEFINER nas funções, geralmente funciona
  • MAS é uma fragilidade - se alguém changer as policies, quebra

⚠️ 7. Tracking de "mês atual" removido sem替代

Impacto: Não há como fazer reset mensal automático.

Das migrações:

-- REMOVIDO: -- mes_atual TEXT DEFAULT to_char(NOW(), 'YYYY-MM'), -- trabalhos_gerados_mes INTEGER DEFAULT 0,

Se no futuro quiser implementar "1 trabalho grátis por mês", não há estrutura para isso. Não há:

  • Coluna para trackar último mês usado
  • Function para resetar monthly
  • Trigger para auto-reset

⚠️ 8. Mensagens de erro não amigáveis

UX: Usuário não sabe o que fazer quando algo dá errado.

Exemplos no código:

  • src/components/FormularioTrabalho.tsx:126 → "Ops! Algo deu errado. Tenta de novo?"
  • src/components/FormularioTrabalho.tsx:84 → "O enunciado deve ter pelo menos 20 caracteres"
  • src/app/api/gerar-pdf/route.ts:123 → "Ocorreu um erro interno ao processar o seu trabalho"

Essas mensagens são genéricas. Usuário não sabe:

  • Se o problema é conexão, API, banco...
  • Se precisa esperar ou tentar novamente
  • Se deve contactar suporte

⚠️ 9. Callback de auth redireciona para /dashboard mesmo em mobile

UX: Pode quebras experiência mobile se não houver redirect correto.

Arquivo: src/app/auth/callback/route.ts:8

const next = searchParams.get("next") ?? "/dashboard";

Sempre redireciona para /dashboard. Em alguns fluxos (ex: veio do landing page), seria melhor voltar para onde estava.

⚠️ 10. Sem verificação de limites no frontend

UX: Usuário só descobre que créditos acabaram DEPOIS de tentar.

No FormularioTrabalho:

  • Usuário preenche tudo
  • Clica em gerar
  • Só AÍ a API retorna erro
  • Então abre modal de pagamento

Melhoria: Mostrar Credits remaining no formulário e desabilitar botão se créditos = 0.
